Инженер данных группы аналитики и автор канала Аналитик на минималках

Кейс: Эффективность маркетинга

Бизнес-запрос:

"Нужен дашборд: рекламные каналы, заказы и выручка. Хотим видеть разрезы по дням, регионам и типам устройств."

Проблемы аналитика:

  • Данные в разных таблицах (visits, orders).
  • Нет единого ключа (Join session_id?).
  • Постоянные пересчеты одних и тех же метрик.

Итог: Медленно, риск ошибок, сложно поддерживать.

Аналитик vs. Инженер данных

Аналитик (SQL)

  • Вопрос "Что?".
  • SELECT / JOIN под отчет.
  • Цель: Визуализация.

Инженер данных

  • Вопрос "Как?".
  • Проектирование продукта.
  • Цель: Устойчивый слой.
"Как сделать так, чтобы запрос писался в одну строку и летал?"

Шаг 1. Определяем "зерно" витрины

Зерно (Grain) — самый детальный уровень данных. 1 строка = ?

1 посещение
1 заказ
1 визит (сессия)

Решение: 1 визит

Позволяет видеть полную воронку. Заказ — это атрибут сессии.

Шаг 2. Ключи и источники

visits orders

Проблема: Нет прямой связи session_id -> order_id

  • visits: session_id, user_id, channel, timestamp...
  • orders: order_id, user_id, revenue, timestamp...
Решение: Сессионизация

Привязываем заказ к последнему визиту пользователя в окне 24 часов. Это фиксированная бизнес-логика в коде ETL.

Шаг 3. Поля и метрики

Измерения (Dimensions)
  • event_date, region, device
  • campaign_id, channel
Факты (Metrics)
  • sessions_count (sum)
  • orders_count, revenue
CREATE TABLE dm_advertising_campaign_performance (
    event_date Date,
    region String, device String,
    -- Метрики
    orders_count UInt8 DEFAULT 0,
    revenue Decimal(10, 2) DEFAULT 0
) ENGINE = MergeTree()
ORDER BY (event_date, region, device);
